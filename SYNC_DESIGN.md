# Синхронизация Notes Vault — устройство

Цель: **все записи (заметки, задачи, теги) автоматически сходятся на всех
устройствах**, где введён один «код синхронизации». При этом сохраняется E2E:
сервер хранит только шифртекст и не может прочитать содержимое.

## Принцип

Синхронизация — это «непрерывный экспорт/импорт через тупое зашифрованное
хранилище». Сервер (Cloudflare Pages Functions) держит по «корзине» набор
элементов и открытую `VaultMeta`; клиент периодически сливает своё состояние с
серверным.

```
устройство A ─┐                       ┌─ устройство B
              ├─►  Pages Functions  ◄─┤
устройство C ─┘   (KV: только ct)     └─ …
```

## Идентичность (без почты) — `core/sync/syncCode.ts`

- **Код синхронизации** — 160-битный случайный секрет (Crockford base32),
  генерируется при включении синхронизации. Это единственный секрет транспорта.
- Из кода детерминированно (BLAKE2b с доменными ключами) выводятся:
  - `syncId` — адрес корзины на сервере;
  - `authToken` — доступ к корзине (Bearer).
- Данные шифруются **ключом волта (DEK)**, не кодом. Новое устройство: ввёл код →
  скачал зашифрованный волт → разблокировал своей кодовой фразой. То есть чтобы
  получить доступ, нужны и код (скачать), и фраза (расшифровать).

## Элемент синхронизации

```
{ collection, id, updatedAt, deleted, ct }
```
`ct` — сериализованный зашифрованный конверт записи (base64). `updatedAt` и
`deleted` — открытые (метаданные), как и в файле экспорта; это принятая моделью
угроз утечка (размер/время/число записей). Содержимое — внутри `ct`, недоступно.

## Слияние — `core/sync/syncEngine.ts` (протестировано)

Last-write-wins по `updatedAt`; ничьи разрешаются детерминированно (надгробие
побеждает, затем сравнение `ct`). `mergeSync(local, remote)` возвращает:
- `merged` — согласованный набор (к нему сходятся клиент и сервер);
- `applyLocal` — что записать/удалить локально;
- `push` — что отправить на сервер.

Операция идемпотентна и сходится независимо от порядка правок на устройствах.

## Сервер — `functions/` (Cloudflare Pages Functions, тот же домен)

Тот же домен, что и приложение → `connect-src 'self'` в CSP работает без правок,
CORS не нужен. Хранилище — Cloudflare **KV** (binding `SYNC_KV`).

- `POST /v1/pull  { syncId }` → `{ meta, items }` (весь снимок корзины).
- `POST /v1/push  { syncId, meta?, items }` → `{ applied }` (LWW-приём).
- Авторизация: `Authorization: Bearer <authToken>`; сервер хранит **хэш** токена
  (trust-on-first-use) и сверяет. Корзины изолированы по `syncId`.

Логика — `functions/_lib/syncStore.ts` (самодостаточна, протестирована на in-memory KV).

## Клиент — `core/sync/syncService.ts`

Цикл `syncNow()` (требует разблокированного волта):
1. Собрать локальные элементы: по манифесту каждой коллекции `{id, updatedAt}` +
   зашифрованный конверт блоба (`ct`) + открытая `VaultMeta` (`meta`).
2. `pull` → удалённые элементы + `meta`.
3. `mergeSync(local, remote)`.
4. Применить `applyLocal`: записать/удалить блобы, при первом заходе на новом
   устройстве записать `VaultMeta`; затем `reindex` затронутых коллекций.
5. `push` итог на сервер.

Запуск автоматический: при разблокировке, после изменений (debounce) и
периодически; плюс ручная кнопка «Синхронизировать сейчас».

## Деплой

1. Cloudflare → **Workers & Pages → KV** → создать namespace, например `notes-vault-sync`.
2. Проект Pages → **Settings → Functions → KV namespace bindings** → добавить
   binding **`SYNC_KV`** → выбранный namespace (для Production и Preview).
3. Пуш в GitHub / `wrangler pages deploy` — `functions/` подхватывается автоматически.
4. В приложении: **Настройки → Синхронизация → Включить** (покажет код) → на
   другом устройстве ввести этот код.

## Ограничения v1 (следующие шаги)

- **Удаления**: движок и сервер уже умеют надгробия; полное распространение
  удалений между устройствами требует локального журнала надгробий (стор в IndexedDB
  + отметка при удалении). В первой версии переносятся создания и изменения.
- Разрешение конфликтов — на уровне записи (LWW), без слияния текста внутри заметки.
- KV `list` — до 1000 ключей за вызов (для личных объёмов достаточно; при росте —
  пагинация уже заложена в `pull`).
