## Часть B. Баги (воспроизведённые дефекты)

### [High] BUG-001: параллельные save теряют запись из манифеста (lost update)
- **Связанный тест:** TC-RACE-01
- **Где:** `src/core/storage/collectionRepository.ts:148-171` (save), `:173-184` (remove)
- **Класс:** CWE-362
- **Статус:** подтверждено прогоном
- **Шаги воспроизведения:**
  1. Создать collectionRepository над общим Repository (fake с задержкой на фазе записи).
  2. Запустить `save({id:A})` и `save({id:B})` параллельно (Promise.all) — оба успевают прочитать манифест до первой записи.
  3. Дождаться обоих; прочитать `listIndex()` и `listBlobIds()`.
- **Ожидаемое:** в индексе обе записи.
- **Фактическое:** блобов 2, в индексе только B; A невидима в списках до reindex.
- **В приложении:** двойной клик по «+ задача» (Главная/Календарь), фоновый save `lastOpenedAt` (NoteDetail.tsx:70) параллельно с действием пользователя, вторая вкладка.
- **Рекомендация:** runExclusive-мьютекс на коллекцию (образец — vaultService.ts:42-50); мультивкладочно — Web Locks API. Дизейбл кнопок на время сохранения.
- **Трудозатраты:** M.

### [Low] BUG-002: счётчик неудач PIN гоняется между вкладками
- **Связанный тест:** TC-VAULT-03
- **Где:** `src/core/vault/vaultService.ts:166-204`
- **Класс:** CWE-362
- **Статус:** подтверждено прогоном
- **Шаги воспроизведения:**
  1. Два экземпляра vaultService над одним Repository (модель двух вкладок); настроить PIN, залочить.
  2. Слать неверные PIN параллельно парами из обоих экземпляров.
- **Ожидаемое:** lockout не позже 5-й суммарной попытки.
- **Фактическое:** lockout после ~10 (потерянные обновления счётчика).
- **Рекомендация:** Web Locks вокруг unlockWithPin или перечитывание meta перед записью счётчика. Помнить: против офлайн-перебора счётчик не защищает вовсе (SEC-01).
- **Трудозатраты:** S.
